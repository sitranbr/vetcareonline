Sistema de Exames Veterinários

Modelagem multi-usuário · multi-tenant · rede de parceiros — alinhado ao SaaS (Abr/2026)

Fonte de verdade: este documento está alinhado ao desenho atual descrito em sass.md (Abr/2026). O modelo foi simplificado para ser legível e fácil de manter: a identidade é do Supabase Auth e o contexto de negócio é definido por profiles + tenants (veterinarians/clinics) + vínculos via profiles.partners e herança via profiles.owner_id.

Nota: blocos mais abaixo que mencionam clinic_memberships, clinic_partnerships e partner_invites são legado (v2). Eles ficam aqui apenas como histórico visual, mas não representam o modelo vigente.

clinics + users▶ 
CREATE TABLE clinics (
  id         UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  name       VARCHAR(150) NOT NULL,
  cnpj       CHAR(14)     UNIQUE NOT NULL,
  phone      VARCHAR(20),
  address    TEXT,
  plan       VARCHAR(20)  DEFAULT 'free',
  status     VARCHAR(20)  DEFAULT 'active',
  created_at TIMESTAMPTZ  DEFAULT now()
);

CREATE TABLE users (
  id             UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  home_clinic_id UUID         REFERENCES clinics(id),  -- nullable: vet autônomo
  name           VARCHAR(100) NOT NULL,
  email          VARCHAR(150) NOT NULL UNIQUE,
  password_hash  TEXT         NOT NULL,
  crmv           VARCHAR(20),
  is_independent BOOLEAN      DEFAULT false,  -- vet sem clínica fixa
  status         VARCHAR(20)  DEFAULT 'active',
  created_at     TIMESTAMPTZ  DEFAULT now()
);
🤝 clinic_memberships▶ 
-- Um usuário pode pertencer a N clínicas com papéis diferentes em cada uma
CREATE TABLE clinic_memberships (
  id        UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id   UUID        NOT NULL REFERENCES users(id),
  clinic_id UUID        NOT NULL REFERENCES clinics(id),
  role      VARCHAR(30) NOT NULL CHECK(role IN (
              'owner', 'staff', 'partner_vet', 'partner_reviewer')),
  status    VARCHAR(20) DEFAULT 'active',
  joined_at TIMESTAMPTZ DEFAULT now(),
  left_at   TIMESTAMPTZ,
  UNIQUE(user_id, clinic_id)
);

CREATE INDEX idx_memberships_user   ON clinic_memberships(user_id)   WHERE status='active';
CREATE INDEX idx_memberships_clinic ON clinic_memberships(clinic_id) WHERE status='active';

-- Quais clínicas esse vet atende hoje?
SELECT c.name, cm.role
FROM  clinic_memberships cm JOIN clinics c ON c.id = cm.clinic_id
WHERE cm.user_id = '<vet-uuid>' AND cm.status = 'active';
🤝 clinic_partnerships▶ 
-- Vínculo formal entre duas clínicas
CREATE TABLE clinic_partnerships (
  id                  UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  requester_clinic_id UUID        NOT NULL REFERENCES clinics(id),
  target_clinic_id    UUID        NOT NULL REFERENCES clinics(id),
  type                VARCHAR(20) NOT NULL CHECK(type IN (
                        'referral', 'lab', 'shared_vet', 'telemedicine')),
  status              VARCHAR(20) DEFAULT 'pending',
  -- O que a clínica-alvo permite à solicitante:
  permissions_json    JSONB       DEFAULT '{}',
  -- ex: {"can_send_orders":true,"can_view_results":true,"shared_vets":["uuid"]}
  accepted_at         TIMESTAMPTZ,
  expires_at          TIMESTAMPTZ,
  created_at          TIMESTAMPTZ DEFAULT now(),
  CHECK(requester_clinic_id != target_clinic_id),
  UNIQUE(requester_clinic_id, target_clinic_id)
);

CREATE TABLE exam_orders (
  id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  requesting_clinic_id UUID        NOT NULL REFERENCES clinics(id), -- quem pediu
  executing_clinic_id  UUID        NOT NULL REFERENCES clinics(id), -- quem processa
  animal_id            UUID        NOT NULL REFERENCES animals(id),
  vet_id               UUID        NOT NULL REFERENCES users(id),

-- Garante que pedido cross-clínica só existe com parceria ativa
CREATE OR REPLACE FUNCTION check_partnership() RETURNS TRIGGER AS $$
BEGIN
  IF NEW.requesting_clinic_id != NEW.executing_clinic_id THEN
    IF NOT EXISTS (
      SELECT 1 FROM clinic_partnerships
      WHERE requester_clinic_id = NEW.requesting_clinic_id
        AND target_clinic_id    = NEW.executing_clinic_id
        AND status = 'active'
        AND (permissions_json->>'can_send_orders')::boolean = true
    ) THEN
      RAISE EXCEPTION 'No active partnership between clinics';
    END IF;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_partnership
  BEFORE INSERT ON exam_orders
  FOR EACH ROW EXECUTE FUNCTION check_partnership();

-- Papel varia por clínica: mesma pessoa pode ser staff na A e partner_reviewer na B
CREATE TABLE user_roles (
  user_id   UUID REFERENCES users(id),
  role_id   UUID REFERENCES roles(id),
  clinic_id UUID REFERENCES clinics(id),
  PRIMARY KEY (user_id, role_id, clinic_id)
);

-- Permissões do usuário em uma clínica específica
SELECT DISTINCT p.resource, p.action
FROM  user_roles ur
JOIN  role_permissions rp ON rp.role_id    = ur.role_id
JOIN  permissions p      ON p.id          = rp.permission_id
WHERE ur.user_id   = '<user-uuid>'
  AND ur.clinic_id = '<clinic-uuid>';
💰 partner_price_lists + partner_price_items▶ 
-- Cabeçalho da tabela de preços negociada entre duas clínicas
CREATE TABLE partner_price_lists (
  id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  seller_clinic_id UUID         NOT NULL REFERENCES clinics(id),  -- lab / executora
  buyer_clinic_id  UUID         REFERENCES clinics(id),           -- null = tabela padrão pública
  name             VARCHAR(100) NOT NULL,
  is_default       BOOLEAN      DEFAULT false,  -- fallback quando não há lista específica
  valid_from       DATE         NOT NULL DEFAULT current_date,
  valid_until      DATE,                          -- null = sem expiração
  status           VARCHAR(20)  DEFAULT 'active'
                   CHECK(status IN ('draft', 'active', 'archived')),
  created_at       TIMESTAMPTZ  DEFAULT now(),
  -- Garante que só existe 1 lista padrão ativa por seller
  UNIQUE NULLS NOT DISTINCT(seller_clinic_id, buyer_clinic_id)
  -- buyer_clinic_id NULL = tabela base/pública do lab
);

-- Itens: um preço por tipo de exame dentro de uma lista
CREATE TABLE partner_price_items (
  id            UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  price_list_id UUID          NOT NULL REFERENCES partner_price_lists(id) ON DELETE CASCADE,
  exam_type_id  UUID          NOT NULL REFERENCES exam_types(id),
  price         DECIMAL(10,2) NOT NULL CHECK(price >= 0),
  discount_pct  DECIMAL(5,2)  CHECK(discount_pct BETWEEN 0 AND 100), -- só informativo
  notes         TEXT,
  UNIQUE(price_list_id, exam_type_id)
);

-- order_items recebe snapshot do preço (imutável mesmo se a lista mudar)
ALTER TABLE order_items
  ADD COLUMN price_list_item_id UUID REFERENCES partner_price_items(id),
  ADD COLUMN unit_price         DECIMAL(10,2);  -- cópia do preço no momento do pedido
💰 Função de resolução de preço▶ 
-- Retorna o preço correto para um exame entre duas clínicas
-- Prioridade: 1º lista específica buyer↔seller  2º lista padrão do seller  3º base_price do exam_type
CREATE OR REPLACE FUNCTION resolve_exam_price(
  p_seller_clinic UUID,
  p_buyer_clinic  UUID,
  p_exam_type     UUID
) RETURNS DECIMAL AS $$
DECLARE
  v_price DECIMAL;
BEGIN
  -- 1. Lista específica para este comprador
  SELECT ppi.price INTO v_price
  FROM  partner_price_items ppi
  JOIN  partner_price_lists ppl ON ppl.id = ppi.price_list_id
  WHERE ppl.seller_clinic_id = p_seller_clinic
    AND ppl.buyer_clinic_id  = p_buyer_clinic
    AND ppi.exam_type_id     = p_exam_type
    AND ppl.status           = 'active'
    AND (ppl.valid_until IS NULL OR ppl.valid_until >= current_date)
  LIMIT 1;

  IF v_price IS NOT NULL THEN RETURN v_price; END IF;

  -- 2. Lista padrão do seller (buyer_clinic_id IS NULL)
  SELECT ppi.price INTO v_price
  FROM  partner_price_items ppi
  JOIN  partner_price_lists ppl ON ppl.id = ppi.price_list_id
  WHERE ppl.seller_clinic_id = p_seller_clinic
    AND ppl.buyer_clinic_id  IS NULL
    AND ppl.is_default       = true
    AND ppi.exam_type_id     = p_exam_type
    AND ppl.status           = 'active'
  LIMIT 1;

  IF v_price IS NOT NULL THEN RETURN v_price; END IF;

  -- 3. Fallback: base_price do tipo de exame
  SELECT base_price INTO v_price FROM exam_types WHERE id = p_exam_type;
  RETURN v_price;
END;
$$ LANGUAGE plpgsql STABLE;

Modelo atual (Abr/2026): parcerias são vínculos bidirecionais entre perfis usando o array profiles.partners. A hierarquia e herança (equipe/guest) usam profiles.owner_id. O objetivo é evitar duplicidade de cadastro e manter o isolamento por tenant com um compartilhamento explícito e auditável.

Conceitos (identidade vs tenant vs vínculos)

🔐 Identidade (Login)

A identidade do usuário é global (Supabase Auth). O sistema hidrata um profile em public.profiles com nível, role e permissões.

  • profiles.id = auth.uid()
  • permissions (JSON) + defaults por nível

🏷️ Tenant (negócio)

O “tenant” de negócio fica em veterinarians ou clinics e aponta para o profile via profile_id.

  • Branding/contato no tenant
  • Dados do sistema referenciam vet/clínica

🤝 Parceiro vinculado

Parceiros são profiles presentes em profiles.partners[] (vínculo mútuo). Isso define visibilidade limitada em RLS e filtros de UI.

  • Vínculo via RPC link_partner_by_email
  • Desvínculo remove de ambos os lados

👥 Equipe / Guest (herança)

Perfis com owner_id diferente de id operam “em nome” do owner: herdam branding e entram em regras de visibilidade como equipe/guest.

  • Recepção/equipe: role=reception ou level=5 + owner
  • Parceiro convidado: vet/clínica com owner do assinante

Contexto de Parceiro (assinante Clínica)

Quando o usuário é assinante do tipo clínica (conta raiz), o Dashboard permite alternar um recorte de visualização “em nome” de um parceiro (um root profile id selecionado). Para consistência, o sistema calcula o conjunto de veterinários pertencentes à árvore do parceiro (fechamento transitivo por profileId/ownerId) e usa isso nos filtros de lista/relatórios.

Regra crítica (evita bug comum): se a clínica logada é uma clínica parceira convidada, o filtro deve considerar o local do exame como a clínica logada (unidade) e o executor como pertencente ao tenant do parceiro/assinante selecionado, sem forçar clinic_id para “a sede” do assinante.

Tipos de parceria suportados

🏥 Clínica → Laboratório

Clínica geral envia pedidos para um laboratório parceiro processar.

  • requesting_clinic ≠ executing_clinic
  • Lab vê apenas os order_items seus
  • Resultado visível à clínica solicitante

🩺 Vet Autônomo Multi-Clínica

Veterinário independente atende em várias clínicas com perfis distintos.

  • home_clinic_id pode ser null
  • clinic_memberships define onde atua
  • user_roles com clinic_id contextual

✍️ Especialista Convidado

Médico veterinário externo convidado para revisar e assinar laudos.

  • Papel: partner_reviewer
  • Acesso restrito a exames designados
  • Rastreado em audit_logs por acting_clinic

🔗 Rede de Referência

Clínica A encaminha pacientes à Clínica B especializada com co-acesso ao histórico.

  • type = 'referral' em clinic_partnerships
  • permissions_json controla visibilidade
  • Pode ter data de expiração

Precificação atual (Abr/2026): a tabela de preços é baseada em regras (ex.: price_rules) por escopo clínica e/ou veterinário, além de modalidade e período. Existem regras “genéricas” (ex.: Todas as Clínicas / Todos os Veterinários) e regras específicas por parceiro.

Regra de segurança: ao filtrar a tabela por um parceiro específico, o sistema não deve incluir regras genéricas (“todos”), evitando vazamento de informação e confusão na UI.

01

Solicitação do Exame

Vet abre pedido definindo requesting_clinic_id e executing_clinic_id. Um trigger valida a parceria ativa. Para cada order_item, a função resolve_exam_price() resolve o preço correto (lista específica → padrão → base) e grava o unit_price como snapshot imutável.

Veterinário Vet Parceiro
02

Roteamento ao Parceiro

Se executing_clinic_id ≠ requesting_clinic_id, o pedido entra na fila da clínica executora. Analistas desta clínica veem apenas os itens filtrados por executing_clinic_id — isolamento garantido por query.

Sistema Lab Parceiro
03

Coleta & Processamento

Analista do lab parceiro registra coleta e lança resultados. O analyst_id pertence à executing_clinic. Audit log registra acting_clinic_id = executing_clinic_id.

Analista (Lab)
04

Assinatura Cross-Clínica

O reviewer_id pode ser um especialista com papel partner_reviewer em ambas as clínicas via clinic_memberships. Sua assinatura é válida se tiver membership ativa na executing_clinic.

Revisor Local Revisor Parceiro
05

Entrega à Clínica Solicitante

Resultado assinado fica visível na requesting_clinic se can_view_results=true na parceria. O vet solicitante recebe notificação e acessa o PDF diretamente.

Recepcionista Veterinário
✦

Auditoria Dupla

Cada ação gera audit_log com o acting_clinic_id correto. Ações da executora ficam no log dela; da solicitante, no log dela. Nenhum dado vaza entre tenants sem rastro.
