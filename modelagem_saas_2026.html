Petcare SaaS

Multi-tenancy + parcerias + white label · Última atualização: Abr/2026

Objetivo: separar identidade (login) do contexto de trabalho (tenant), mantendo isolamento por RLS e permitindo compartilhamento explícito via parcerias. Este documento é um “mapa mental” único do modelo descrito em sass.md (Abr/2026).

Princípios:
1) identidade global em auth.users
2) perfil e regras em public.profiles (role, level, owner_id, permissions, partners[])
3) dados de negócio em tenants (veterinarians / clinics)
4) parcerias via profiles.partners (vínculo mútuo)
5) herança via profiles.owner_id (equipe/guest herda do assinante)

Identidade, Perfil e Tenant

profiles → veterinarians veterinarians.profile_id
profiles → clinics clinics.profile_id

Equipe / Guest / Herança

profiles → profiles owner_id aponta p/ perfil do assinante

Parcerias (vínculo mútuo)

profiles ↔ profiles partners[] contém IDs (texto) em ambos os lados

Exames, contexto e white label

exams → clinics clinic_id = local
exams → veterinarians veterinarian_id = executor
Isolamento: RLS no Supabase/Postgres define quem vê/edita cada linha. Operações críticas usam RPC SECURITY DEFINER com search_path protegido para evitar injeção e loops em políticas.
Essa visibilidade também alimenta a UI: o dropdown e os filtros de parceiros resolvem equipe via profiles.partners/profiles.owner_id e então listam exames por exams.clinic_id/exams.veterinarian_id.
Policies (visão)▶
-- Visualização de exames (conceito)
-- exams.veterinarian_id e exams.clinic_id são TEXT (UUID como texto)
ALLOW SELECT on exams IF
  exam_visible_to_tenant(clinic_id, veterinarian_id)

-- Visualização de perfis da equipe do parceiro
-- Evita recursão usando função SECURITY DEFINER (anti-loop na leitura de profiles)
ALLOW SELECT on profiles IF
  id = auth.uid()
  OR owner_id = auth.uid()
  OR (owner_id IS NOT NULL AND profile_owner_is_in_my_partners(owner_id))

Smart Linking: evita duplicidade de cadastro. Ao invés de “Clínica A criar Dr. João” e “Clínica B criar Dr. João” (dois logins), o sistema cria um vínculo entre perfis existentes via profiles.partners.

Fluxo de vínculo (email → parceria)

01

Busca prévia

Procura o e-mail nas tabelas profiles, veterinarians e clinics para identificar se já existe (nome/role/id).

02

RPC link_partner_by_email

Função SECURITY DEFINER aplica o vínculo mútuo em partners[] após validar: existe, não auto-vínculo, sem duplicidade.

03

Desvincular

Remove IDs de ambos os arrays partners[], mantendo consistência bidirecional.

04

Fallback “Guest”

Se o usuário não existe, pode-se criar um convidado com owner_id apontando para o criador. Esse perfil pode ser “emancipado” futuramente.

Gestão de equipe vs parceiro

CategoriaComo identificarPrincipal efeito
Membro interno profiles.owner_id = tenant_profile_id herda branding; visibilidade definida por owner_id
Parceiro vinculado profileId em partners[] visibilidade limitada; aparece em seletores/listas
Parceiro convidado vet/clínica com owner_id do assinante opera “em nome” do assinante; importante para filtros
Tabela de preços dinâmica: regras por clínica e por parceiro, com modalidades e períodos. O sistema suporta modalidades personalizadas e taxas extras.

Nota de manutenção: ao filtrar por parceiro específico, evite incluir regras genéricas (“todos”) na listagem de preços.

Estrutura da regra (PriceRule)

PriceRule (shape)▶
type PriceRule = {
  id: string
  clinicId: string         // escopo (clínica)

  modality: Modality

  repasseClinic: number

}

01

Autenticação & Sessão

Ao iniciar: verifica sessão no Supabase Auth, cria usuário temporário a partir do user_metadata, define tenant provisório e hidrata o profile completo em profiles.

02

Tenant & Contexto

Carrega tenant por profile_id; se faltar, tenta por e-mail e vincula. Mantém currentTenant e availableTenants. Para assinante clínica, pode haver modo “contexto de parceiro” por root profile.

03

White Label (Branding efetivo)

SettingsContext busca branding do tenant do usuário. Se equipe/recepção ou guest, herda do owner: resolve tabela (vet vs clinic) pelo role do owner e carrega dados por profile_id.

04

Exames & Laudos

Exame contém dados do paciente, requisitante (nome/CRMV), vínculos (executor, clínica local, dono da máquina), modalidade/período e valores (total/repasses). O editor gera PDF com white label.

05

Preços & Repasses

Valores são calculados com base nas regras de preço (modalidade/período/escopo) e no dono da máquina; suporta taxas extras e split automático entre profissional e clínica.
